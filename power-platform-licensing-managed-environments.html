Power Platform Licensing & Managed Environments

A plain-language overview of how Power Platform is licensed, and what changes when you turn on Managed Environments. Prepared as an enablement reference · July 2026.

The one thing to remember: A Managed environment is a premium capability. Once you manage an environment, every user with active usage in it must hold a standalone premium license (Power Apps Premium, Power Automate Premium, Copilot Studio, Power Pages, or a qualifying Dynamics 365 plan) — or the environment must have qualifying pay-as-you-go meters / per-app capacity allocated.

1. The core distinction: “seeded” vs “premium”

Most licensing confusion comes down to this split. What a user can build and run depends on which bucket their license falls into.

Seeded (bundled)

Included with Microsoft 365 / Dynamics 365

  • Build apps & flows on Microsoft 365 data and standard connectors only.
  • No premium/custom connectors.
  • No on-premises data.
  • D365-seeded rights apply only within the context of that Dynamics 365 app.
Premium (standalone)

Power Apps / Power Automate Premium, Copilot Studio, etc.

  • Premium & custom connectors.
  • On-premises data gateway access.
  • Create Dataverse in non-default (production/sandbox) environments.
  • Includes Managed Environments as an entitlement.

3. What “Managed Environments” actually gives you

Managed Environments are a suite of premium governance capabilities that let admins manage Power Platform at scale — more control, less effort, more insight. You can apply it to any environment type, and it’s toggled per environment (not all-or-nothing across the tenant).

4. The licensing rule for Managed Environments

When you enable Managed Environments in an environment, all active usage requires a qualifying standalone license or PAYG meter. In practice:

  • Every user running an app in a managed environment needs Power Apps Premium, Power Automate Premium, or a qualifying Dynamics 365 Enterprise license (or per-app capacity allocated to the environment).
  • Every user running a cloud flow in a managed environment needs the same (or Flow per-business-process capacity allocated).
  • A single user who runs both an app and a flow does not need two licenses — either Power Apps Premium or Power Automate Premium satisfies the requirement.
  • Flows triggered within an app or agent are covered by that app’s / Copilot Studio license.
  • Trial licenses can cover users in managed environments, but trial restrictions (e.g. 30-day) still apply.
The trade-off: Managed Environments = ease of administration & governance vs. the cost of premium licensing for the users in those environments. A common pattern: leave the default environment unmanaged for citizen makers (locked-down DLP, no premium), and reserve managed environments for a smaller community of enterprise builders who hold premium licenses.

5. Per-user vs per-app: which is cheaper?

Rule of thumb: if a user actively uses ~10 or more apps, a per-user license is usually the better value. Below that, per-app licensing (allocated to the environment) tends to win. Model both against your actual usage — the break-even shifts with your mix.

6. Copilot Studio note

Copilot Studio has its own consumption model (Copilot Credits / PAYG) layered on top of environment licensing. Key nuance for agents:

  • An M365 Copilot-licensed user interacting with an agent in an authenticated chat burns no Copilot Studio credits — their license covers it.
  • Autonomous agents (triggered by schedule, email, or a file drop) have no license coverage and always consume Copilot Studio credits.
  • A Copilot Studio agent requires a Dataverse-enabled environment.
