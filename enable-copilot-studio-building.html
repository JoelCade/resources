Microsoft Copilot Studio · Enablement Reference

Enabling Copilot Studio Building for an Organization

The two supported ways to give makers the right to build and manage agents in Microsoft Copilot Studio — the seat-based path and the pay-as-you-go path. Prepared as an enablement reference · July 2026.

The one thing to remember: There are two ways to entitle someone to build agents — and you should always weigh both. Either (1) buy the seat-based licenses (a tenant license plus a per-user Copilot Studio User License), or (2) enable pay-as-you-go on an environment and grant the maker the Environment Maker role. Runtime consumption is metered as Copilot Credits either way.

The two enablement paths at a glance

Building an agent requires a build entitlement. That entitlement can come from a purchased seat, or from a pay-as-you-go environment plus the right security role. Neither path is "more correct" — pick the one that fits the org's procurement and usage profile.

Path 1 · Seat-based

Tenant license + per-user license

  • Predictable, planned rollouts.
  • Entitlement lives on the user's seat.
  • Purchased in the Microsoft 365 admin center (or via volume licensing).
Path 2 · Pay-as-you-go

PAYG environment + Environment Maker

  • Dev/test, spiky usage, "give me a sub and let me build."
  • Entitlement lives on the environment, not the seat.
  • Billed directly to an Azure subscription.

Path 1 — Seat-based licensing

Two distinct licenses are required — one at the tenant level and one per builder.

1

Tenant license (organization-level)

Listed as "Copilot Studio" in the Microsoft 365 admin center. The tenant administrator acquires it once — it cannot be assigned to individual users. It enables Copilot Studio for the tenant and backs the message / Copilot Credit consumption pool that agents draw from at runtime.

2

Per-user license (builder-level)

Listed as the "Copilot Studio User License" in the admin center. Assigned to each individual who needs to create and manage agents. Anyone building, editing, or managing agents needs one.

Volume-licensing rule: If you buy through volume licensing (EA/CSP) or any channel other than the Microsoft 365 admin center, you must acquire both the tenant license and the user licenses through that same channel — preferably in a single transaction.

Path 2 — Pay-as-you-go + Environment Maker

This path decouples building from prepaid seats entirely. Instead of buying a Copilot Studio User License per maker, you attach an environment to Azure billing and grant makers the environment role that carries build rights.

1

Enable pay-as-you-go on the environment

In the Power Platform Admin Center (PPAC), the admin creates a billing policy that links the target environment to an Azure subscription. Copilot Studio consumption (and the underlying Power Platform requests / Copilot Credits) is then metered and billed directly to Azure — no prepaid tenant capacity or per-user Copilot Studio seat required.

2

Assign the Environment Maker role

Grant each maker the Environment Maker role on that pay-as-you-go environment. With Maker rights in a PAYG-enabled environment, a user can create and manage agents in Copilot Studio without holding a Copilot Studio User License — the build entitlement comes from the environment/PAYG configuration rather than the seat.

Why teams choose this: No seat procurement, no upfront commitment, and spend follows actual usage. It's the natural fit for pilots, hackathons, dev/test environments, and organizations that would rather point a maker community at an Azure subscription than pre-provision seats.

Who does not need a license

  • End users of a published agent — no special license required; anyone with access to the published agent can interact with it.
  • Guest users of the tenant — cannot access Copilot Studio at all, under either path.

Runtime consumption — common to both paths

Regardless of how you entitle building, agents consume Copilot Credits at runtime. The seat path draws them from prepaid capacity; the PAYG path meters them to Azure. A few nuances worth knowing:

  • An M365 Copilot-licensed user interacting with an agent in an authenticated chat typically burns no Copilot Studio credits — their license covers it.
  • Autonomous agents (triggered by schedule, email, or a file drop) have no license coverage and always consume Copilot Studio credits.
  • A Copilot Studio agent requires a Dataverse-enabled environment.
  • Forecast an agent's credit volume with the public Copilot Studio agent usage estimator before you deploy.
