Modern Copilot Studio: The New Orchestrator On-Ramp

A builder's guide to the new agent-and-workflow paradigm in Copilot Studio — what changed, what to build, and how to bring classic agents forward. Understand it, see it, upgrade it. · July 2026.

The one thing to remember: Every behavior belongs in the smallest component that makes it reliable and inspectable. Instructions carry what's always true, Knowledge the searchable facts, Tools the system actions, Memory the persistent context, Skills the situational procedures, and connected agents the real specialist domains. A modern agent shouldn't be one instruction blob with 43 tools and a prayer.

Why this matters

The new Copilot Studio experience — and the orchestration stack underneath it — is a genuine shift in how you design agents and automation. It's a new paradigm on two fronts: agents are far more adaptive and sophisticated, and workflows let you build automated processes on a visual canvas with much more control over which steps are handled by AI. That's a lot of new capability, and it changes the questions you ask: What actually changed? What do I build, and where? And what happens to my existing classic agents?

There are three resources built to answer exactly those questions — a deck, a mini-site, and a plugin. This page explains how to use each one, and folds in the core concepts from the Technical Deep Dive so you can get oriented before you open any of them.

Agents

Mostly adaptive

Reason over context and decide the next step at runtime, where the path can't be mapped up front — and call deterministic steps when part of the work is fixed.

Workflows

Mostly deterministic

Run a defined sequence the same way every time, where the process is predictable and repeatable — and call agents when part of the work needs reasoning.

They are not an either-or choice. Enterprise solutions combine both; the real question is which part of a process should carry the load deterministically and which should reason adaptively, and how the two work together.

1. Understand it: the concepts behind the deck

The deck reframes the move from classic orchestration to modern agents. Classic builds a plan, then runs it. The new orchestrator keeps deciding from the latest state — a continuous Thought → Action → Observation loop rather than a fixed plan-then-execute. In Microsoft's testing, the new orchestrator showed roughly 20% higher evaluation accuracy with about 50% fewer tokens.

What modern agents do that classic couldn't

CapabilityWhat changed
Asking fewer, smarter questionsInstead of a rigid prompt for every tool input in a fixed order, modern agents infer what's already known, ask only what's missing, and can bundle or reorder questions.
Handling detoursA side question or mid-task change no longer terminates the plan — the agent answers, then picks the task back up.
Orchestrating over toolsBetter tool selection, chaining one tool's output into the next, and running independent tools in parallel.
Following instructionsInstruction-first, not built bottom-up from tool descriptions. Write reusable skills in markdown — or import existing GitHub Copilot / Claude Code skills — and expect them to work.
Recovering from errorsWhere classic stops and surfaces an error, a modern agent may retry intelligently or take an alternative path to still finish.
Flowing across turnsQuestions, tool calls, and responses interleave fluidly instead of the rigid plan-then-execute pattern.

The building blocks of a modern agent

The core design rule: choose the smallest component that makes the behavior reliable, inspectable, and safe inside the loop.

Instructions

Role, tone, what not to say, and how the agent drives the conversation. What's true in every conversation. Keep them lean — everything here loads on every turn.

Knowledge

Searchable docs, sites, and files used to ground answers.

Tools / MCP servers

Live lookups, APIs, and CRUD operations — how the agent acts on systems.

Memory

Persistent context across sessions: user preferences and history.

Code interpreter

A full Python runtime in an isolated container for data manipulation, file analysis, and generation — on by default, no single-file limit, charts render everywhere.

Skills & connected agents

Skills package situational procedures (a SKILL.md plus bundled resources) loaded on demand. Connected agents delegate real specialist sub-domains.

Skills: instructions on demand

A Skill is just instructions and resources, loaded on demand — keep it scenario-specific, name it precisely, and test that it fires. The metadata (name + description) sits in context and drives routing; the full content loads only when a scenario matches.

What to avoid with Skills: vague descriptions (weak routing signals make a Skill mis-fire or never trigger), mega-skills (broad do-everything scope blurs intent), fact dumps (a Skill guides — it doesn't replace a Knowledge source), and unreviewed Skills (treat AI or community Skills as untrusted code and review them first).

2. See it: the mini-site and a real sample

The mini-site is built around BlastBox Omega, a retro-future game store run by agents. It shows the new experience doing the things a slide can only promise: agents that reason across multiple turns, delegate to specialists, take real actions, and produce actual deliverables. Two scenarios make that concrete:

Self-Serve Card Reissue

An agent handles a member request end to end — gating a real write action behind an identity check and handing back a generated file.

Block Party Trade-Up

The flagship: a parent agent coordinates specialist agents to untangle a messy, multi-part request and settle it with a downloadable document.

The value is seeing where each responsibility lives: specialist reasoning in connected agents, actions in tools, repeatable procedures in Skills, and exact math in code. Read the building blocks, run the scenario transcripts, then download and deploy the solution into your own Power Platform environment.

3. Upgrade it: the migration plugin

Install the Copilot Studio plugin for AI coding agents, then send /migrate with the agent's environment, tenant, and Copilot Studio URL plus any constraints. It clones the classic agent locally, inspects it, computes a target architecture using Microsoft's own best practices, and generates an upgraded agent you can fine-tune rather than rebuild from scratch.

Starter prompt (use a capable AI model):

/mcs-assistant:migrate Upgrade this agent to modern orchestration:
https://copilotstudio.microsoft.com/environments/<ENV_ID>/bots/<BOT_ID>
from tenant <TENANT_ID>
The key word is propose. It's a fast assistant, not a "make my architecture correct" button. Don't turn every topic into a Skill and every variable into memory just because they existed — that's archaeology with YAML. Treat the output as a first draft: run it, inspect it, compare against your old evals, and decide if it's good enough.

Migrate capabilities, not components

The migration mental model is re-architect, don't port. Ask what tasks the agent must accomplish, then map each responsibility to a loop-native component:

Classic constructModern target(s)
Topics (conversational paths)Instructions for flow; a Skill for a reusable procedure; a Tool when the topic mainly performs one action or lookup.
Variables (stored state)Short-lived context in the loop; memory for user context; Dataverse for durable task context; code interpreter if the value is derived.
Power Fx (deterministic logic)Explicit code in a tool or skill helper file.
Adaptive CardsActive work underway on a rich-UI framework (see gaps below).
Agent flowsWorkflows with native AI actions and agent hand-offs.

A simple loop works well: understand the use case and its evals → inventory the existing agent → refactor tasks (not components) into loop-native blocks → evaluate old vs. new on the core journeys, and repeat until evals pass. The migration is done when the agent can perform its core tasks — not when every old artifact has been mechanically copied.

Agents vs. workflows: choosing the foundation

Yesterday the choice was binary — a fully autonomous agent that decides everything, or classic automation on rails. Today there's a controllable middle: workflows keep the deterministic reliability of agent flows and bring in agent-node reasoning (or human review) only where judgment is needed.

QuestionAgent-firstGoal-first (workflow)
Needs AI at all?Reasoning everywhere by defaultYou choose where AI earns its place
Who decides the steps?The orchestrator works it out at runtimeYou author the path yourself
A step needs judgment?The whole process is one agentAdd reasoning on just that step
Which model runs it?One model must cover the hardest stepEach step picks the model that fits

Rule of thumb: the decision comes down to how much of the process is known before runtime. Workflows suit processes where start and end points are known with variability at the step level (e.g. expense approval); agents suit broad problem spaces that can't be fully mapped up front (e.g. a multi-turn healthcare intake dialogue). Workflows can call agents, and agents can call workflows.

An honest read: gaps and workarounds

Most classic authoring and conversation logic maps cleanly onto instructions, skills, and tools. A few cases need a workaround or are on the roadmap:

  • UI components — rich cards and structured input capture (the classic Adaptive Cards role) are the main true gap today; a suitable framework is being identified.
  • System / environment variables — no direct equivalent; instruct the agent to call a tool that fetches user context before it responds.
  • Message intercept hooks — no general intercept point yet (classic "On message received" / "On AI response generated"); hooks are planned.
  • Grounded-knowledge-only mode — by design not offered; modern agents have the autonomy to decide when to generate versus ask.
  • Builder analytics — work underway to enable both the classic dashboards and the new experience.
